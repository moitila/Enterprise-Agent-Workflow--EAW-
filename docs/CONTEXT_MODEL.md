# Context Model Contract

## Purpose

This document defines the canonical context model for EAW. It separates stable repository context from runtime-derived operational context and records the contract boundaries needed for auditability.

## Context Types

### onboarding

`onboarding` is stable context maintained for the repository, stored outside the target repository, and consumed by EAW.
It changes rarely and is intended to describe long-lived repository facts, conventions, and operating constraints.

### dynamic_context

`dynamic_context` is operational context generated by the runtime from the card.
It changes by phase and is limited to selecting and delimiting evidence.
It must not interpret evidence, conclude, replace findings, or introduce analytical judgment.

## Materialization Principle

Context only exists for the system when it is materialized as an observable and auditable artifact.
If a context element cannot be inspected in artifact form, it is not considered available for governed execution.

## Physical Location of Context Sources

- `onboarding` source: `<EAW_WORKDIR>/context_sources/onboarding/<repo_key>/`
- `onboarding` materialized: `out/<CARD>/context/onboarding/`
- `dynamic_context` materialized: `out/<CARD>/context/dynamic/`
- `templates`: `templates/context/<type>/<template_name>/`, versioned through `ACTIVE`

## Boundary: `context` vs `tooling_hints`

- `tooling_hints` are operational instructions to the agent about how to execute the phase.
- `context` is a declaration of the context that must be collected and injected into the prompt.
- Workspace-sourced onboarding context and EAW template rendering are separate concerns.
- `phase.yaml` must not use free-form contextual prose outside the approved contract fields.

## Traceability

Every injected context artifact must identify:

- its source (`onboarding`, `dynamic`, or `template`)
- the template applied
- the source artifact location under `out/<CARD>/context/onboarding/` or `out/<CARD>/context/dynamic/`

The injected context must be readable by an engineer, not only by the agent. If a prompt artifact cannot explain what was injected and why, the audit trail is incomplete.

## Responsibilities

- Repository owners and EAW maintain the workspace-sourced onboarding context outside the target repository.
- The runtime derives dynamic context from the current card and phase.
- The EAW template layer renders and presents context without redefining it.

## Non-Goals

- Implementing collectors or parsers.
- Changing workflow YAML schemas.
- Expanding the runtime context engine contract beyond this model.
